# ベンチマーク設計メモ

## 現在の計測方法

**計測 API:** `performance.now()`
ミリ秒単位（小数点以下含む）の高精度タイマー。

**計測している範囲（何が含まれているか）:**

```
JS 計測
└─ applyFiltersJS(originalImageData, filters)
   ├─ Uint8ClampedArray のコピー生成
   ├─ 各フィルター処理（brightness / contrast / blur など）
   └─ new ImageData() で結果生成

WASM 計測
└─ applyFilters(wasmModule, originalImageData, filters)
   ├─ JS→WASM メモリへのコピー (wasmMem.set(...))
   ├─ 各フィルター処理（WASM 関数呼び出し）
   └─ WASM メモリ→JS への読み出し (result.data.set(...))
```

**計測回数:** 1回のみ

---

## 現状の問題点

| 問題 | 影響 |
|---|---|
| **1回計測のみ** | GC・スケジューラの割り込みなど偶然の外れ値に結果が左右される |
| **ウォームアップなし** | JS エンジンの JIT コンパイルが走る前の遅い状態を計測してしまう可能性がある |
| **JS → WASM の順番固定** | JS を先に実行することで CPU キャッシュが温まり、WASM が有利になる可能性がある |
| **WASM 計測にメモリコピーが含まれる** | 純粋な演算速度だけでなく JS⇔WASM 間のデータ転送コストも込みの数字になっている |

現在の数字は「ページに表示された画像に対して今のフィルター設定を1回処理したときの経過時間」であり、統計的な信頼性はない。

---

## 信頼性を上げる改善案

### 1. ウォームアップ

JIT が効いた状態にするため、計測前に数回実行する。

```ts
// ウォームアップ（結果は捨てる）
for (let i = 0; i < 3; i++) {
  applyFiltersJS(originalImageData, filters);
  applyFilters(wasmModule, originalImageData, filters);
}
```

### 2. 複数回計測して統計値を取る

```ts
const RUNS = 10;
const jsTimes: number[] = [];
const wasmTimes: number[] = [];

for (let i = 0; i < RUNS; i++) {
  const t0 = performance.now();
  applyFiltersJS(originalImageData, filters);
  jsTimes.push(performance.now() - t0);

  const t1 = performance.now();
  applyFilters(wasmModule, originalImageData, filters);
  wasmTimes.push(performance.now() - t1);
}

// 中央値を採用（外れ値に強い）
const median = (arr: number[]) => {
  const sorted = [...arr].sort((a, b) => a - b);
  return sorted[Math.floor(sorted.length / 2)];
};
```

### 3. 実行順のバイアスを除去

JS → WASM 固定ではなく、交互に実行してキャッシュの影響を均す。

```ts
for (let i = 0; i < RUNS; i++) {
  if (i % 2 === 0) {
    // JS → WASM
    measureJS(); measureWasm();
  } else {
    // WASM → JS
    measureWasm(); measureJS();
  }
}
```

### 4. メモリコピーを分離して計測（オプション）

WASM の「純粋な演算時間」と「JS⇔WASM 転送コスト」を分けて計測することで、どこがボトルネックかを明確にできる。

| 項目 | 説明 |
|---|---|
| JS 中央値 | 10回計測の中央値 (ms) |
| WASM 中央値 | 10回計測の中央値 (ms) |
| 速度差 | JS中央値 / WASM中央値 |
| 画素数 | width × height (px) |
| スループット | px/ms（画素数 ÷ 処理時間） |

## アーキテクチャ上の2つの問題

ベンチマーク数値が期待より小さい（1.2〜1.3倍程度）根本原因として、以下の2つの問題がある。

---

### 問題A：JS↔WASM メモリコピーのオーバーヘッド

**影響範囲:** WASM モードのみ

スライダーを1つ動かすたびに、画像データ全体を2回コピーしている。

```
スライダー変更
  ↓
[JS メモリ: 4MB] ──コピー──→ [WASM メモリ: 4MB]
                                      ↓
                                   処理実行
                                      ↓
[JS メモリ: 4MB] ←──コピー── [WASM メモリ: 4MB]
```

WASM の演算自体は速いが、この転送コストが優位性を打ち消している。
計測結果は「演算時間 + 転送時間」の合計になっており、WASM の本来の速さが見えていない。

**改善案:** オリジナル画像を WASM メモリに常駐させる

```
画像アップロード時（1回だけ）:
  JS メモリ → WASM メモリ[オリジナル領域]

スライダー変更ごと:
  WASM 内部コピー → [作業領域] → 処理 → JS メモリ（canvas 描画用）
  ↑ JS↔WASM 転送ではなく WASM 内の memcpy なので高速
```

WASM メモリのレイアウト（改善後）:

```
| オリジナル (0〜N) | 作業バッファ (N〜2N) | ブラー用 (2N〜3N) |
       ↑ 変更しない        ↑ ここで処理
```

削減できるもの: アップロード時の1回を除き、JS→WASM のコピー（約4MB）が毎回不要になる
残るもの: 結果を canvas に描画するための WASM→JS コピーは避けられない

---

### 問題B：フィルターパイプラインの全再計算

**影響範囲:** JS・WASM 両方

1つのスライダーを動かしただけで、有効なフィルター全てをオリジナルから再計算している。

```
明るさだけ変えたのに...

オリジナル → 明るさ → コントラスト → 彩度 → ブラー
               ↑          ↑           ↑       ↑
           変えた！   変えてない   変えてない  変えてない
                      なのに再計算  なのに再計算 なのに再計算
```

フィルター数が増えるほど無駄な計算量が増える。
現在の設計はオリジナルから毎回やり直すことで画質劣化を防ぐ「非破壊編集」であり、
この再計算は意図的な設計だが、パフォーマンス面では非効率。

**改善案A（破壊編集）:** 各フィルターを順番に確定させ、前の結果に積み上げる

```
明るさ確定 → コントラスト確定 → 彩度確定 → ...
    ↑前の結果に上書きしていく（速い）
```

デメリット: 一度確定したフィルターを後から変更できない、Undo/Redo が壊れる

**改善案B（パイプラインキャッシュ）:** 各ステップの結果をキャッシュし、変更点以降だけ再計算する

```
オリジナル → [明るさ結果をキャッシュ] → [コントラスト結果をキャッシュ] → ...
                       ↑
               コントラストだけ変わったら
               ここから再計算すればよい（上流はスキップ）
```

メリット: 速さと非破壊編集を両立
デメリット: 各ステップ分の ImageData をメモリに保持する必要がある、実装が複雑

---

## どちらを先に対応すべきか

| | 問題A（メモリコピー） | 問題B（全再計算） |
|---|---|---|
| 影響範囲 | WASM のみ | JS・WASM 両方 |
| フィルターが少ない場合 | 効果大 | 効果小 |
| フィルターが多い場合 | 効果中 | 効果大 |
| 実装の複雑さ | 中 | 中〜大 |
| ベンチマーク数値への影響 | 大（WASM 本来の速さが見える） | 中 |
| アプリの操作感への影響 | 小 | 大（スライダーが軽くなる） |

**「JS vs WASM の差を正確に計測したい」なら問題A を先に対応する。**
**「アプリ全体を速くしたい」なら問題B（改善案B）を先に対応する。**
